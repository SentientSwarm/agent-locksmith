## 4. Common point of confusion: "no LLM proxy endpoints"

The M0-era SAP design sketched separate `/llm/*`, `/mcp/*`, and `/a2a/*`
endpoint namespaces. **v2 deliberately removed them.** The current
router exposes exactly one proxied path:

```
ANY /api/{tool_name}/{*path}    → proxy::proxy_handler
```

LLMs are configured as ordinary tool entries (`anthropic`, `openai`,
`lmstudio`, etc.) and reached via `/api/<name>/...`. The reasoning
chain in PRD:

- **D-11** Composability over absorption — Locksmith does not implement
  egress controls (Pipelock), prompt-injection scanning (LlamaFirewall),
  or LLM gateway functionality (Kamiwaza, LiteLLM).
- **D-15** Egress routing is per-tool, not per-model or per-payload.
- **D-17** Wire-format-agnostic — new providers are YAML, not Rust.
- **R-F18** No payload inspection.

So "Locksmith does not have proxy endpoints for LLMs" is *literally*
true (no `/llm/*` namespace) but **misleading** — Locksmith proxies LLM
traffic with full credential injection, mTLS, audit, response-size cap,
redaction, and Pipelock egress, just under `/api/<tool>/...`. The M1
streaming work (`tests/streaming_passthrough_test.rs`,
`inference_matrix_*` tests) was specifically to make this path safe for
SSE and multi-minute generations.

## 5. LlamaFirewall positioning (D-18)

`PRD.md` §D-18 is explicit: cognitive scanning libraries (LlamaFirewall,
NeMo Guardrails, Guardrails AI) are **library calls inside the agent
process**, invoked during the reasoning loop on inputs/tool-results/
outputs. Locksmith operates at the network credential boundary. They
**compose by both being present, in parallel, not in series on the
wire.**

The team's earlier work that wrapped LlamaFirewall in its own HTTP
proxy with credential handling and identity is exactly the
"cognitive-scanner-on-the-wire anti-pattern" `SPEC.md §3028` flags as a
risk. Locksmith does **not** see scanner verdicts and does not stamp
its own audit on scanner activity.

D-18 corollary: when middleware *does* sit between Locksmith and an
upstream, Locksmith treats it as **an ordinary tool entry** — Locksmith
injects credentials into it; the middleware forwards. The middleware
does not hold credentials of its own, does not implement its own agent
identity, does not duplicate Locksmith's audit.

### Two ways the FastAPI / docker-compose sidecar can fit

1. **Compatible today, no Locksmith change.** Run a FastAPI sidecar
   that imports the LlamaFirewall library and exposes whatever HTTP
   surface the agent needs. Configure the sidecar as an ordinary
   Locksmith tool entry: agents call `/api/<sidecar>/...`, Locksmith
   injects credentials, sidecar performs its in-process scan and
   returns. This is the D-18 corollary path. Pipelock + Locksmith +
   the sidecar in one docker-compose stack is straightforward.

2. **Inline inspector — post-v2, design-gated.** Locksmith calls the
   sidecar in the hot path between agent and upstream with
   pass/drop/modify verdicts, streaming awareness, and audit emission.
   This is **issue #82** and is explicitly listed as `1 design + 1
   impl` of post-v2 work. It re-litigates D-18 under controlled
   conditions and is **not started**. If you want pattern (2),
   that has to begin with a written design under `docs/v2/` covering
   wire/socket protocol, fail-open vs fail-closed, audit event shape,
   and how it composes with M7 streaming caps.

If your current goal is "compose Locksmith + Pipelock + a LlamaFirewall
sidecar in docker-compose," pattern (1) is achievable on `develop`
today with zero Locksmith code changes — just YAML.

## 6. What should *not* be ported forward

When reading the deprecated top-level `SPEC.md`, treat the following as
**explicitly rejected by v2**, not as future work:

- The `/llm/{provider_name}/{path}`, `/mcp/{server_name}/{path}`, and
  `/a2a/{path}` endpoint namespaces.
- Locksmith calling a "scanner sidecar" inline before/after upstream
  forwarding (subsumed by D-18; only #82 is the design slot).
- Per-provider code paths, Anthropic-version header injection, etc.
  Provider compatibility is YAML, not Rust (D-17).
- `cloud: bool` as a tool field — replaced by `egress: direct |
  proxied` (M1, with one-shot deprecation warning via the
  DeprecationRegistry).
- `sap` / `secure-agent-proxy` naming — binaries are `locksmithd` and
  `locksmith`.

## 7. Quick orientation for code review

- Composition root: `src/daemon.rs::run` — read this first.
- Router + AppState: `src/app.rs::build_app_full`.
- Hot path: `src/proxy.rs::proxy_handler` (header strip, credential
  inject, egress select, response controls, audit emit).
- Auth dispatch: `src/auth.rs::auth_middleware` stamps `AuthenticatedAs`
  + `AgentIdentity` into request extensions.
- mTLS: `src/mtls/{validator,crl,blocklist,authenticator}.rs`.
- Admin substrate: `src/admin/{service,uds,https,bootstrap_listener}.rs`
  + `src/repo/{agent,bootstrap,audit}.rs`.
- Secrets: `src/secret/{mod,env,file_sealed,vault,aws}.rs`.
- Response controls: `src/response_controls.rs`.
- CI: `cargo test --all`, `cargo clippy --all-targets -- -D warnings`,
  `cargo fmt --check`.
- DB migrations are embedded in the binary; `migrations::open_and_migrate`
  applies them on daemon start.
